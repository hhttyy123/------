# 劳务派遣经营管理系统技术实施方案

状态：当前有效  
更新日期：2026-07-04

## 1. 实施目标

在保留当前 Excel 预览、数据集 CRUD 和查询能力的基础上，将系统逐步升级为支持多用户、关系型数据、审批联动、自动预警和可追溯核对的劳务派遣经营管理系统。

实施主线：

```text
数据收集与清洗
→ 关系型数据库存储
→ 确定性业务规则处理
→ AI 顾问解释与辅助判断
→ 审计、监控和备份贯穿全程
```

原则：先让数据可靠进入系统，再建设复杂业务功能；先让程序算对事实，再让 AI 解释事实。

## 2. 总体架构

### 2.1 技术选型

- 前端：React、TypeScript、Vite、Tailwind CSS、shadcn/ui。
- 后端：FastAPI、Pydantic、SQLAlchemy 2.x、Alembic。
- 数据库：PostgreSQL；本地开发可以使用 Docker PostgreSQL。
- 文件存储：开发环境本地目录，生产环境使用兼容 S3 的对象存储。
- 异步任务：第一阶段使用数据库任务表和后端任务执行器；任务量增加后再引入 Redis/Celery 或同类组件。
- AI：通过可替换的模型适配层调用，不能成为核心流程的硬依赖。
- 部署：前后端联网部署，HTTPS 访问，数据库和文件定期备份。

### 2.2 服务边界

```text
前端工作台
  ├─ 首页 / 预警 / 待办
  ├─ 人员与用工
  ├─ 企业与合作
  ├─ 财务与薪资
  ├─ AI 顾问
  └─ 设置
        ↓
FastAPI 应用
  ├─ 身份认证与权限
  ├─ Excel 导入导出
  ├─ 业务模块 API
  ├─ 审批与预警规则
  ├─ 联动与核对服务
  ├─ 报表服务
  └─ AI 顾问服务
        ↓
PostgreSQL + 文件存储 + 审计日志
```

初期保持一个模块化单体应用，不拆微服务。当前团队和业务规模下，模块化单体更容易保证事务一致性和开发效率。

## 3. 第一支柱：数据收集与清洗

### 3.1 标准导入流程

每次导入分为七个阶段：

1. 上传：校验扩展名、文件大小、病毒/恶意内容风险，生成文件哈希。
2. 扫描：读取 Sheet、行列数、合并单元格和候选表头。
3. 映射：优先使用模块模板和已保存映射；无法确定时由用户手工确认。
4. 标准化：转换日期、金额、手机号、身份证、枚举和空值。
5. 校验：执行必填、唯一、关联、范围和跨字段规则。
6. 预览：展示新增、更新、重复、警告和阻断错误。
7. 提交：用户确认后，在数据库事务中批量写入，并记录批次和来源。

AI 不参与决定字段映射或自动修改原始值。后续可以让 AI 解释错误原因，但程序和用户仍是最终决策者。

### 3.2 导入的三层数据

- 原始层：保存上传文件、Sheet、行号和原始单元格内容，作为证据。
- 暂存层：保存标准化结果、字段映射和校验问题，供用户确认。
- 业务层：只写入通过校验并被确认的数据。

三层分离可以避免错误清洗直接污染正式数据，也允许重新应用新规则。

### 3.3 幂等与重复检测

- 文件级：使用文件 SHA-256、文件名和大小识别完全重复上传。
- 行级：根据模块业务键生成导入指纹。
- 人员示例：身份证号优先；无身份证时使用姓名、手机号等组合并提示人工确认。
- 工资示例：人员、工资月份、发薪批次和实发金额。
- 回款示例：企业、回款日期、金额、付款方式和银行流水号。
- 用户可选择跳过重复、更新已有记录或作为新记录导入。

### 3.4 模板与专项适配

建立导入适配器接口：

- 通用标准模板适配器。
- 人员、企业、考勤、工资、返费、开票和回款适配器。
- `2026 年曼克斯日记账` 专项适配器。
- `苏州曼克斯 2026 年利润核算表` 专项适配器。

日记账适配器将并排的收入区、支出区和月度汇总区解析为独立数据区域：

```text
收入区域 → 标准日记账收入记录
支出区域 → 标准日记账支出记录
月度汇总 → 校验数据，不混入明细
```

### 3.5 导入结果模型

每条暂存记录至少保存：

- `import_batch_id`
- `source_file_id`
- `source_sheet`
- `source_row`
- `source_region`
- `raw_data`
- `normalized_data`
- `record_fingerprint`
- `status`
- `issues`

## 4. 第二支柱：数据库设计

### 4.1 数据建模原则

- PostgreSQL 是正式数据的唯一权威来源。
- 业务数据不再以 JSON 文件作为生产存储。
- 同一业务事实只保存一次，其他模块通过关联查询或派生记录使用。
- 金额使用 `NUMERIC`，禁止浮点误差。
- 所有业务表具有创建人、创建时间、修改人、修改时间和版本号。
- 重要财务记录原则上不物理删除，使用作废和冲正。
- 所有关键联动在同一数据库事务中完成。

### 4.2 基础与权限表

- `users`：用户账号和状态。
- `roles`：一级员工、二级财务、三级老板及管理员。
- `permissions`：模块、数据范围、字段和操作权限。
- `user_roles`、`role_permissions`：角色关联。
- `audit_logs`：不可变操作日志。
- `attachments`：照片、合同和资质文件元数据。
- `system_settings`：逾期阈值、预警天数和业务参数。

### 4.3 人员与用工表

- `employees`：人员档案。
- `employment_records`：入职、调动和离职历史。
- `attendance_records`：每日考勤和工时。
- `contracts`：员工合同、企业合同和派遣协议。
- `contract_events`：签订、续签、终止等过程记录。

身份证号在数据库中加密存储，另存不可逆哈希用于唯一性判断。

### 4.4 企业与合作表

- `companies`：企业档案和合作状态。
- `positions`：企业岗位、单价和岗位描述。
- `company_qualifications`：营业执照等资质及有效期。
- `company_daily_stats`：可选的每日聚合缓存；原始事实仍来自入职和工资数据。

照片和资质文件保存在对象存储，数据库只保存附件关联和权限信息。

### 4.5 财务与薪资表

- `payroll_batches`：发薪批次。
- `payroll_items`：个人工资明细。
- `recruitment_rebates`：代招返费。
- `invoices`：开票记录。
- `receivables`：企业应收记录及到期日。
- `payments`：实际回款。
- `payment_allocations`：回款与发票/应收的多对多分配。
- `cash_transactions`：统一资金收支和日记账事实表。
- `transaction_links`：资金流水与工资、返费、回款等来源记录关联。
- `profit_periods`：已确认的期间利润快照。
- `profit_lines`：利润构成及来源追溯。
- `reconciliation_items`：核对差异和处理状态。

### 4.6 审批与预警表

- `approval_requests`：审批单及当前状态。
- `approval_steps`：每级提交、审核、确认和驳回记录。
- `alerts`：预警实例。
- `alert_rules`：可配置阈值和启停状态。
- `alert_actions`：预警处理过程。

### 4.7 导入导出表

- `files`：上传文件元数据和哈希。
- `import_batches`：导入批次。
- `import_mappings`：确认过的模板和字段映射。
- `import_rows`：原始、标准化和校验结果。
- `export_jobs`：导出任务、筛选条件和文件结果。

## 5. 第三支柱：数据处理

### 5.1 确定性规则引擎

规则以普通 Python 服务实现，规则输入和输出结构化，并具备独立测试。第一阶段不引入复杂规则平台。

适合确定性规则的内容：

- 利润、工资、返费和日结工薪资计算。
- 上班超过 20 天未签合同。
- 合同到期前 N 天。
- 回款逾期天数和逾期金额。
- 银行承兑到期提醒。
- 工资、返费、回款生成资金流水。
- 业务记录与日记账匹配核对。
- 三级审批状态流转。
- 权限判断。

每条规则需要：规则编号、版本、启用时间、输入字段、输出结果和自动化测试。利润等重要规则变更后，历史期间结果不能静默变化；应保留规则版本并支持重新计算对比。

### 5.2 业务联动方式

工资、返费和回款确认时不复制一份无法追踪的日记账数据，而是创建带来源关联的资金流水。

示例：

```text
工资审批最终确认
→ 更新工资状态
→ 创建资金支出 cash_transaction
→ 创建 transaction_link(source_type=payroll, source_id=...)
→ 写操作日志
→ 触发利润期间重新计算标记
→ 同一事务提交
```

修改已确认记录时，优先使用更正单或冲正记录。若业务允许直接修改，则必须重新审批，并在同一事务中更新关联流水。

### 5.3 预警计算

- 实时规则：业务保存或状态变化时立即检查。
- 定时规则：每天固定时间扫描合同、回款、承兑和资质到期情况。
- 去重：同一对象、同一规则只保留一个未解决预警。
- 关闭：条件恢复正常时自动关闭，或由有权限用户处理关闭。

预警事实由程序计算；AI 只解释影响并给出处理建议。

### 5.4 利润核算

实现前先确认利润口径。系统设计支持：

- 现金口径：以实际回款和实际支出为准。
- 权责口径：以已确认收入、成本和应收为准。

计算服务按期间读取被确认的收入和成本事实，生成利润明细及来源关联。任何汇总数字必须可以下钻到工资、返费、回款或其他收支原记录。

## 6. AI 顾问实施方式

### 6.1 上下文管道

1. 用户选择问题涉及的数据集、模块或时间范围。
2. 后端检查权限。
3. 后端生成结构化查询计划并执行查询。
4. 后端将聚合结果和必要的有限明细交给模型。
5. 模型解释结果、指出异常并提出建议。
6. 回答展示数据范围、时间和引用的系统记录。

### 6.2 安全边界

- 模型不直接连接数据库。
- 模型不能绕过后端权限。
- 默认不发送身份证、手机号等敏感信息。
- AI 输出与事实计算分开展示。
- AI 请求失败时返回可恢复错误，不影响其他功能。
- AI 建议的修改必须转为待确认操作，不能直接写库。

### 6.3 第一批顾问能力

- 解释首页预警和核对差异。
- 总结月度工资、返费、回款和利润变化。
- 回答企业、人员、月份维度的自然语言查询。
- 给出逾期回款跟进建议。

## 7. API 与模块组织

后端建议按业务模块拆分：

```text
app/
  core/          # 配置、数据库、鉴权、权限、异常
  imports/       # 上传、扫描、映射、清洗、校验、提交
  employees/
  companies/
  attendance/
  contracts/
  payroll/
  finance/
  approvals/
  alerts/
  reports/
  advisor/
  audit/
```

API 使用版本前缀 `/api/v2`。旧 `/api/v1` 在迁移期间保留，只修复阻断性问题，不继续扩展新业务。

关键 API 类型：

- `/auth/*`：登录和用户信息。
- `/imports/*`：上传、预览、校验和提交。
- `/employees/*`、`/companies/*`：业务 CRUD 和关联查询。
- `/payroll/*`、`/rebates/*`、`/invoices/*`、`/payments/*`。
- `/journal/*`、`/profit/*`、`/reconciliation/*`。
- `/approvals/*`、`/alerts/*`。
- `/reports/*`：查询、导出和打印。
- `/advisor/chat`：受权限控制的顾问问答。

## 8. 前端实施

### 8.1 页面框架

- 顶部：当前页面、全局搜索、通知、用户菜单。
- 左侧：三个可折叠业务分组、AI 顾问和设置。
- 主区：指标、筛选器、表格和详情抽屉。
- 首页：指标、预警、待办、核对异常和最近操作。

### 8.2 通用表格能力

保留并重构当前数据表格能力：

- 服务端分页、排序和筛选。
- 列显示、固定、宽度和顺序配置。
- 按人员、企业和时间快速筛选。
- 批量选择和批量操作。
- 当前筛选结果导出 Excel。
- 大表虚拟滚动。
- 字段类型对应正确的编辑控件和校验。

### 8.3 日记账页面

- 收入、支出、月度汇总三个页签。
- 收入表只显示收入字段和非空记录。
- 支出表只显示支出字段和非空记录。
- 月度汇总与明细分离。
- 支持日期、月份、年度、类别和来源筛选。
- 每条记录可以打开来源业务记录和核对状态。

## 9. 分阶段交付

### 阶段 0：安全基线与现状冻结

预计目标：建立可回退的开发基线。

- 清理 `.gitignore`，排除真实数据、上传文件、日志和密钥。
- 保存脱敏测试样本。
- 创建首个 Git 提交。
- 将当前 18 个后端测试和前端 build/lint 纳入基础检查。

### 阶段 1：导入清洗 MVP

交付：

- PostgreSQL、SQLAlchemy 和 Alembic 基础。
- 文件、导入批次、暂存行和映射表。
- 标准导入七阶段流程。
- 人员与 2026 日记账两个首批适配器。
- 日记账收入/支出/汇总分开展示。
- 错误行下载和重复导入检测。

验收：同一文件重复导入不产生重复数据；每条记录可追溯到原始行。

### 阶段 2：人员、企业与合同

- 用户登录和基础角色权限。
- 人员、企业、岗位、入离职和合同。
- 合同未签、合同到期预警。
- Excel 导入导出和操作日志。

### 阶段 3：工资、返费与三级审批

- 考勤和日结工资基础。
- 工资批次、个人明细和代招返费。
- 三级审批状态机。
- 确认数据自动生成资金流水。
- 日、月、年支出统计。

### 阶段 4：开票、回款、日记账与核对

- 开票、应收、回款和银行承兑。
- 回款逾期预警和跟进记录。
- 日记账业务联动和外部流水导入。
- 自动匹配与核对中心。

### 阶段 5：利润和专项利润表

- 确认利润口径。
- 月、季度和年度利润计算。
- 利润构成下钻。
- 苏州曼克斯 2026 年利润表导入、对照和导出。

### 阶段 6：AI 顾问与增强功能

- AI 顾问受控查询和解释。
- 员工个人薪资查看。
- 企业资质预警。
- 自定义报表和打印。
- 电子签名等外部能力评估。

## 10. 测试策略

- 导入适配器：真实结构脱敏样本的固定回归测试。
- 清洗函数：日期、金额、身份证、手机号的参数化测试。
- 数据库：唯一约束、外键、事务和并发测试。
- 规则：利润、逾期、合同预警和联动的单元测试。
- API：权限、审批状态和错误契约测试。
- 前端：导入、筛选、编辑、审批和日记账页签的组件/端到端测试。
- Excel：导入后导出再读取的往返测试。
- AI：验证权限边界、敏感字段过滤和服务失败降级；不以模型文案完全一致作为测试条件。

## 11. 数据迁移

当前 JSON 数据和通用数据集不能直接视为正式生产数据。迁移步骤：

1. 备份原始 JSON 和 Excel。
2. 生成数据质量报告。
3. 通过新的导入暂存层重新清洗。
4. 用户确认映射和异常记录。
5. 写入 PostgreSQL。
6. 对记录数量、金额合计和关键月份进行对账。
7. 迁移验收后，旧存储改为只读归档。

## 12. 第一轮具体任务

第一轮只聚焦“数据进入系统是否可靠”：

1. 建立安全 Git 基线和脱敏样本。
2. 接入 PostgreSQL、SQLAlchemy 和 Alembic。
3. 实现导入批次、原始行、标准化行和校验问题模型。
4. 抽取可复用的清洗、校验和字段映射框架。
5. 完成 2026 年现金日记账适配器。
6. 前端实现收入、支出、月度汇总分开展示。
7. 实现重复导入检测、错误行导出和来源追溯。
8. 用现有真实表格的脱敏副本建立回归测试。

完成这轮后，再进入人员和企业数据库建模，避免在不可靠的数据入口上继续堆业务功能。
